\section{Introduction}
In this project, we built a standalone preferential crawler for the MedOverNet domain (\url{https://medover.zurnal24.si/}). The goal is to crawl relevant pages first, respect robots rules, avoid duplicate content, and store crawl data in PostgreSQL using the required crawldb schema.
% NOTE (code sync): Current runtime is websocket daemon mode only (standalone mode is disabled in pa1/crawler/src/main.py).

The crawler is designed as a multi-worker system with a database-backed frontier. Each worker takes one URL, checks robots and politeness limits, downloads or renders the page, detects duplicates, stores the result, extracts new links and images, scores discovered URLs, and pushes them back to the frontier.

\section{System architecture and implementation}
\subsection{High-level pipeline}
The crawler processes each URL in the following order:
\begin{enumerate}
    \item Claim next URL from the frontier.
    \item If the canonical URL already exists in \texttt{crawldb.page} as \texttt{HTML/BINARY/DUPLICATE}, do not re-download; only insert the \texttt{link} record (from the source page to the already-known target).
    \item Check robots rules and enforce politeness delay.
    \item Download content (or render with headless browser when needed).
    \item Detect exact duplicates using content hash.
    \item Store page data and metadata in crawldb.
    \item Parse outgoing links and images.
    \item Score discovered URLs and enqueue them into frontier.
\end{enumerate}

\subsection{Main components}
The implementation is split into clear modules:
\begin{itemize}
    \item Worker orchestration and crawl loop: \texttt{scheduler.py}, \texttt{run\_crawler.py}.
    \item Frontier and page persistence: \texttt{frontier\_store.py}, \texttt{page\_store.py}.
    \item Downloader and renderer: \texttt{downloader.py}, \texttt{renderer.py}.

    \item Robots and politeness: \texttt{robots.py}, \texttt{politeness.py}.

\subsection{Frontier queue semantics}
The URL frontier is implemented in a dedicated table, \texttt{crawldb.frontier\_queue}, rather than in \texttt{crawldb.page}.
Each frontier row stores a canonical absolute \texttt{url}, \texttt{priority}, \texttt{source\_url}, and \texttt{depth}, with a uniqueness constraint on \texttt{url}.

Enqueue uses UPSERT semantics and is concurrency-safe: if the URL already exists, priority is merged with \texttt{GREATEST},
depth/discovery metadata are kept from the best available values, and previously terminal rows (\texttt{COMPLETED}/\texttt{FAILED}) can be reactivated to \texttt{QUEUED}.

Workers claim URLs via an atomic transaction: candidates are read in priority order
(\texttt{ORDER BY priority DESC, discovered\_at ASC}) with \texttt{FOR UPDATE SKIP LOCKED}, then the selected row is moved to \texttt{LOCKED}
and tagged with lease metadata (\texttt{locked\_at}, \texttt{locked\_by\_worker\_id}). Expired leases are automatically re-queued after a configurable TTL.

Queue state is enum-driven (\texttt{QUEUED}, \texttt{LOCKED}, \texttt{PROCESSING}, \texttt{COMPLETED}, \texttt{DUPLICATE}, \texttt{FAILED}).
Completion reports map worker outcomes to terminal states, while retry outcomes map back to \texttt{QUEUED}. Politeness timing is enforced server-side through
in-memory per-daemon/per-site cooldowns (minimum 5\,s), not through a \texttt{frontier\_next\_fetch\_time} column.

\section{Preferential crawling strategy}
The crawler uses a preferential strategy: every discovered URL is assigned a numeric priority (stored as \texttt{priority}
in \texttt{crawldb.frontier\_queue}), and workers always claim the currently due URL with the highest priority.
% NOTE (code sync): Queue column name is \texttt{priority} in \texttt{crawldb.frontier_queue} (not \texttt{frontier_priority}).

\subsection{Scoring function}
The current implementation uses a lightweight, configurable heuristic (\texttt{relevance.py}).
For a discovered URL \texttt{u} with parent URL \texttt{p} at depth \texttt{d}, we compute:
\begin{itemize}
    \item \textbf{Same-host boost:} if \texttt{host(u) == host(p)}, add \texttt{same\_host\_boost} (default: 10).
    \item \textbf{Allowed domain suffix boost (optional):} if the host matches any configured suffix, add \texttt{allowed\_suffix\_boost} (default: 20).
    \item \textbf{Keyword boosts (optional):} for each configured keyword occurring in the URL path/query, add \texttt{keyword\_boost} (default: 5).
    \item \textbf{Depth penalty:} subtract \texttt{depth\_penalty} $\cdot d$ (default: 0.2).
\end{itemize}

With the default configuration, the crawler primarily prefers same-host URLs and mildly prefers shallow depth. Domain-specific focusing is supported via configuration, but must be explicitly set.
% NOTE (code sync): Current defaults also include non-empty topic keywords (medicine, health, doctor, clinic, hospital, treatment, disease), so keyword boosts are active by default.

\subsection{Worked example with default parameters}
For a child URL discovered on the same host at depth $d=3$:
\[
\text{score} = 10 - 0.2\cdot 3 = 9.4
\]
For an off-host URL at the same depth (no same-host boost):
\[\text{score} = -0.6\]
Therefore, same-host pages are explored first unless additional boosts are configured.

\section{Robots.txt and politeness}
The crawler enforces robots and ethics constraints before each fetch:
\begin{itemize}
    \item robots fetched per origin and cached,
    \item \texttt{Allow}/\texttt{Disallow} checked for crawler user-agent,
    \item \texttt{Crawl-delay} read when present,
    \item \texttt{Sitemap} URLs collected for optional frontier seeding,
    \item hard minimum delay of 5 seconds per target IP, even if robots does not define delay.
\end{itemize}

The implementation uses a thread-safe per-IP limiter. Effective delay is:
\[
    \text{effective\_delay} = \max(5s, \text{robots\_crawl\_delay})
\]

This design allows parallel workers while still respecting per-target pacing.

\section{Downloader, renderer, and content-type handling}
\subsection{HTTP downloader}
The downloader uses \texttt{requests} with crawler user-agent and timeout settings from configuration. It follows redirects and stores status code and final URL.

\subsection{JS rendering}
For pages where static HTML is not enough, the crawler can render using headless Chrome via Selenium. This mode is optional because it is slower, and we use it when needed for dynamic content.

\subsection{Content-type policy}
The crawler distinguishes HTML and binary resources:
\begin{itemize}
    \item HTML: stored in \texttt{page.html\_content}, \texttt{page\_type = HTML}
    \item PDF: supported as optional binary download, otherwise metadata-only
    \item DOC/DOCX/PPT/PPTX: metadata listed in \texttt{page\_data}, no binary payload required
    \item other non-HTML resources: marked as \texttt{BINARY}
\end{itemize}

In this assignment crawl HTML content by default, handle PDF where needed, and avoid storing unnecessary binary blobs.

\section{Deduplication}
\subsection{Exact deduplication}
The crawler computes a content hash (SHA-256 of the exact HTML source) for HTML pages. The ingestion logic distinguishes:
\begin{itemize}
    \item \textbf{Known canonical URL:} if the canonical URL already exists in \texttt{crawldb.page}, the crawler does not insert a new page row and does not treat this as a duplicate; it only records the hyperlink in \texttt{crawldb.link}.
    \item \textbf{Duplicate content:} if the canonical URL is new but the content hash matches an existing \texttt{HTML} page, a new row is inserted with \texttt{page\_type\_code = DUPLICATE}.
\end{itemize}

If the content hash already exists:
\begin{itemize}
    \item new page is marked \texttt{DUPLICATE},
    \item \texttt{duplicate\_of} points to the original \texttt{HTML} page,
    \item duplicate page does not store full HTML content.
\end{itemize}

\section{Problems and solutions}
\subsection{Robots and ethical crawling}
\textbf{Problem:} parallel workers can violate politeness if delays are not shared.\\
\textbf{Solution:} server-side per IP locking and queue delagation, the shared queue holds timeouts for the next allowed fetch of each origin, when a worker requests a deque, and the server returns the highest priority URL that is not timed out, and adds it to the timeout list until the next allowed fetch time.
This design allows multiple workers to operate in parallel while still respecting per-origin crawl delays. The new problem is that workers on different IPs are rate limited by the same delay, but this is a necessary tradeoff for ethical crawling. The other easy solution would just be to just do proxy switching, but this is not ethical.

\subsection{Dynamic content}
\textbf{Problem:} some links/content are injected by JavaScript.\\
\textbf{Solution:} optional Selenium rendering path for selected pages.

\subsection{Frontier concurrency}
\textbf{Problem:} multiple workers may claim the same URL.\\
\textbf{Solution:} The workers don't connect to the database directly, this is also a security design to allow workers to run on seperate machines without exposing the database to the networ. Instead the frontier is implemented on the server. When a worker is in a \textit{idle} state, the server deques the highest priority URL, and locks it by updating it's status in the database, this removes it from the frontier candidates. To ensure that the queue is fast and responsive, the server maintains an in-memory queue of the urls in the fronntier. The databae table is also indexed by the frontier\_priority and frontier\_next\_fetch\_time, so that the server can quickly find the next available URL to assign to a worker. If a worker fails to fetch a URL, the server can re-enqueue it by updating its status and frontier\_next\_fetch\_time in the database.
% NOTE (code sync): Current DB claim path uses \texttt{FOR UPDATE SKIP LOCKED} and lease fields (\texttt{locked_at}, \texttt{locked_by_worker_id}) in \texttt{crawldb.frontier_queue}.
% NOTE (code sync): Current primary queue index is state + priority DESC + discovered_at ASC; \texttt{frontier_next_fetch_time} is not part of the schema.

\subsection{Duplicate handling}
\textbf{Problem:} same content may appear under different URLs.\\
\textbf{Solution:} exact hash check and \texttt{DUPLICATE} page type with pointer to original.

\subsection{Spider traps and loops}
\textbf{Problem:} repeated navigation patterns and low-value loops can waste crawl budget.\\
\textbf{Solution:} canonical URL uniqueness in DB, relevance scoring, and robots compliance constraints.
